2. The client's intent

The client runs multiple physical clinic locations. They want the platform to model each clinic as a branch, with these constraints:

  1. Master super admin creates branch-admin accounts. One branch = one admin. Branch metadata (name, address, phone, GSTIN, UPI ID) is captured inline in the same form — no separate "Branches" CRUD UI.
  2. The super admin is view-only across every screen except the Branch Admins page. They select a branch from a top-bar dropdown and see exactly what that branch's admin would see — bookings, payments, patients, doctors, services, pincodes, analytics — but cannot create / edit / delete anything.
  3. Branch admins (created by the super admin) keep the full current admin toolkit, but everything they see is automatically scoped to their branch.
  4. Patients pick a branch as Step 0 of the booking flow before they see the existing patient-info / address-and-slot / confirm-and-pay wizard.
  5. The currently demo'd admin@gmail.com account "is basically one branch admin" — convert him to the Main Branch admin so the existing demo keeps working unchanged.
  6. Super-admin dashboard shows a branches-comparison widget on the "All branches" view, and the analytics page gets a revenue-by-branch bar chart + pie chart + share table.
  7. One new table (branches) is unavoidable; everything else uses branch_id columns on existing tables.
  8. The live demo URL must NOT show this feature until Phase 2 ships. Strategy: apply the DB migration now (Phase 2 capable), but keep the new code on a separate branch until cutover.

6.3 Data mutation

UPDATE users
   SET role       = 'admin',
       admin_role = 'admin',
       branch_id  = 1
 WHERE email      = 'admin@gmail.com'
   AND role       = 'super_admin';

Idempotent (the WHERE clause excludes the row after the flip succeeds).

6.4 New indexes

  • branches_active_idx on branches(is_active) WHERE is_active = TRUE
  • idx_bookings_branch_status on bookings(branch_id, booking_status, scheduled_date DESC)
  • idx_pincodes_branch on serviceable_pincodes(branch_id)
  • idx_doctor_centers_branch on doctor_centers(branch_id)

6.5 Intentionally NOT modified

  • clinic_settings — its UNIQUE(key) constraint would conflict with per-branch overrides. Per-branch UPI / GSTIN / phone live on the branches table directly; clinic_settings remains the global fallback.
  • uq_active_slot on bookings — not changed; a doctor cannot be at two branches at the same hour anyway.
  • serviceable_pincodes.pincode UNIQUE — kept global for now; if two branches need to share a pincode, promote to (pincode, branch_id) in a follow-up migration.

7. Architecture — what was built (code side)

7.1 Backend (Express on Node, pg, JWT, bcrypt)

  • JWT payload (backend/src/lib/jwt.ts): added optional branch_id?: number | null.
  • Auth middleware (backend/src/middleware/auth.ts): AuthUser now carries branch_id; new helpers:
    • requireSuperAdmin — sugar over requireRole('super_admin').
    • scopedBranchFilter(req) — returns the branch_id to filter admin reads by. admin → pinned to own; super_admin → reads ?branch_id=N or NULL (= no filter).
    • enforceBranchAdminWrite — rejects super_admin writes with 403 SUPER_ADMIN_READ_ONLY.
  • Auth routes (authRoutes.ts): login / signup / OTP responses include branch_id; /auth/me selects it too.
  • New router branchAdminsRoutes.ts — mounted under /api/v1 in server.ts. Endpoints listed in section 9.
  • Router-level write gate in phase2Routes.ts — intercepts every POST | PATCH | PUT | DELETE under /admin/*, runs requireAuth then enforceBranchAdminWrite. Branch-admin router is mounted separately so it bypasses this gate.
  • Booking POSTs: /bookings/test, /bookings/doctor-appointment, walk-in admin endpoint. Each accepts optional branch_id via Zod, defaults to 1 at the application layer (defense in depth on top of the DB DEFAULT 1).
  • Admin read endpoints branch-scoped via scopedBranchFilter:
    • /admin/bookings — WHERE clause
    • /admin/bookings/:id — cross-branch attempts return 404
    • /admin/dashboard — KPIs scoped; extra payload branches_comparison[] for super_admin "All branches"
    • /admin/analytics — trends + breakdowns scoped; extra payload revenue_by_branch[] for super_admin "All branches"
    • /admin/serviceable-pincodes — WHERE clause
  • /serviceable-pincodes/check — public endpoint, accepts optional ?branch_id.

14. Rollback plan

14.1 Pre-Phase-2 (currently)

If anything looks off right now (it shouldn't — verified clean), the DB can be rolled back manually:

BEGIN;
ALTER TABLE bookings ALTER COLUMN branch_id DROP NOT NULL;
ALTER TABLE bookings ALTER COLUMN branch_id DROP DEFAULT;
-- Reverse the role flip:
UPDATE users SET role='super_admin', admin_role='super_admin', branch_id=NULL
 WHERE email='admin@gmail.com';
-- Optional: drop the new columns and table entirely (NOT NEEDED for Phase-1 safety):
ALTER TABLE users DROP COLUMN branch_id;
ALTER TABLE bookings DROP COLUMN branch_id;
ALTER TABLE doctor_centers DROP COLUMN branch_id;
ALTER TABLE serviceable_pincodes DROP COLUMN branch_id;
DROP TABLE branches CASCADE;
COMMIT;

Easier: restore the Supabase backup taken before the migration ran.

14.2 Post-Phase-2

If a problem surfaces after merging Phase 2 into main:

git revert -m 1 <merge-sha>
git push origin main

Phase 1 code redeploys, and because the schema is backward-compatible (DEFAULT 1 on bookings.branch_id), the old code keeps working against the migrated DB.

15. Do NOT do this

  1. Do NOT run pnpm db:seed against the demo DB while Phase 1 is still deployed. The seed in main still inserts admin@gmail.com as super_admin and would flip his role back. Functionally that's fine (both roles still pass guards in Phase 1 code), but it's inconsistent with the migration's intent. Wait until after the Phase-2 merge.
  2. Do NOT delete or rename the branches table or any branch_id column. The deployed Phase-1 code doesn't reference them, but the new code expects them on day one.
  3. Do NOT remove the DEFAULT 1 on bookings.branch_id until after Phase 2 ships. It's the single guardrail that keeps Phase-1 booking INSERTs working.
  4. Do NOT force-push to main. Use a normal merge.
  5. Do NOT seed superadmin@gmail.com manually before Phase-2 ships — the deployed Phase-1 frontend has no UI for them and they'd just sit unused.
  6. Do NOT touch clinic_settings. Per-branch overrides live on the branches table itself. clinic_settings is left as the global default fallback.
  7. Do NOT cherry-pick the migration to main before merging the Phase-2 branch. The migration is already applied to the DB; only the code part is pending.
