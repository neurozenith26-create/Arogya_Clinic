01 · About the product

What this app is, who uses it, what problems it solves.

Product summary

Arogya Diagnostics & Multispeciality Clinic (Kolkata) operates a public marketing site, a patient portal, an admin operations panel, and a walk-in billing surface — all from one codebase.

A single patient identity walks through: discover services → book → pay (manual UPI for now) → get sample taken (in-clinic or at home) → wait → download reports → download invoice. Admins run the entire back office: catalog, doctors + schedules, slot generation, walk-in bills, payment re-verification, home-collection dispatch, report uploads, analytics, audit log.

Three audiences, one repo

  • Visitors — phase-1 marketing site, contact form, public reviews.
  • Patients — sign up, book, pay, view reports, download invoices.
  • Admin / super_admin — operations, catalog, walk-in, reports, analytics.

What makes it tick

  • booking_code ties patient → booking → items → payments → invoice → reports.
  • One transaction creates booking + items + invoice + payment row.
  • Live slots generated from doctor schedule, never stored.
  • Backend-owned authz — RLS dropped, JWT re-validated per request.

What is still in flight

  • SMS OTP via MSG91 — currently logs debug_code in non-prod.
  • Razorpay signature verification + real checkout — currently a stub.
  • Slot-lock expiry cron for stale drafts.
  • S3 storage driver — falls back to local writes.

02 · Run it locally (zero to running app)

From an empty checkout to a working dev environment in ~5 minutes.

Prerequisites

Node 20+ pnpm 9+ Postgres 14+ (or Supabase) macOS / Linux / Windows (PowerShell)

1) Install

# at repo root
pnpm install

2) Backend env (backend/.env)

DATABASE_URL=postgres://user:pass@localhost:5432/arogya
JWT_SECRET=replace-with-16-or-more-chars-secret
PORT=3001
NODE_ENV=development
TZ=Asia/Kolkata
CORS_ORIGIN=http://localhost:5173
DATABASE_SSL=false
JWT_EXPIRES_IN=7d
BCRYPT_ROUNDS=10
OTP_EXPIRY_MINUTES=10
OTP_MAX_ATTEMPTS=5
STORAGE_DRIVER=local
STORAGE_LOCAL_PATH=./uploads
STORAGE_PUBLIC_BASE_URL=http://localhost:3001/uploads
# optional: MSG91_*, RAZORPAY_*, RESEND_API_KEY, SENTRY_DSN

3) Frontend env (frontend/.env)

VITE_API_URL=http://localhost:3001/api/v1
VITE_PHASE_2_ENABLED=true
VITE_USE_MOCK_DATA=false

4) Migrate + seed

pnpm db:migrate   # applies supabase/migrations/0001..0026
pnpm db:seed      # categories, departments, services, demo accounts

5) Run dev (both apps in parallel)

pnpm dev          # turbo dev — frontend 5173, backend 3001
# or individually:
pnpm dev:frontend
pnpm dev:backend

6) Open the apps

  • Patient site — http://localhost:5173
  • Patient login — /auth/login
  • Admin login — /admin/login
  • Health check — /api/v1/health

07 · Payment system — every mode, every state

Three sources × five states × five methods. Tables show the exact column transitions.

payment_source enum (what kind of payment row)

payment_sourceWhen usedRequired fields
upi_manualPatient checkout — UPI QR + uploaded screenshot/PDF proofpayment_proof_bytes + mime, verified_at=NULL until admin re-verifies
offlineWalk-in bill + admin recording in-person paymentcollected_by_user_id (the staff who took the money)

Mode A — Manual UPI (current default)

Patient scans QR / pays UPI ID, uploads a screenshot/PDF + optional UTR. Slot locks immediately. Admin re-verifies in person at the clinic / at sample collection.

Stagebookings.booking_statusbookings.payment_statuspayments.payment_statuspayments.verified_at
Patient submits proof at /book/*in_progresspartial / paidcapturedNULL
Admin opens Re-Verify modalin_progresspartial / paidcapturedNULL (yet)
Admin clicks Re-verifyconfirmedpartial / paidcapturedNOW()

Idempotent: CTE only fires when verified_at IS NULL; a second click returns the already-verified row.

Mode B — Offline (walk-in & in-person top-ups)

Receptionist captures payment with the walk-in bill, or records a top-up against an existing booking. collected_by_user_id is mandatory so we know who took the cash.

methodContext
cashWalk-in patient hands over cash at reception
upi_qr_offlineStatic QR scan at reception (no online verification yet)
card_swipePOS terminal at reception
chequeManual capture, cleared later (no auto-clearing logic)

After insert, the handler takes a FOR UPDATE lock on the booking row and recomputes paid_amount / balance_amount / payment_status from SUM(amount − refunded_amount) WHERE payment_status IN (captured, authorized).

Mode C — Razorpay (stub, for future)

Wiring is preserved so the project can switch back when the clinic activates a merchant account.

  • POST /payments/create-order currently inserts a row with razorpay_order_id='order_mock_<ts>'.
  • POST /webhooks/razorpay logs into webhook_events with ON CONFLICT (event_id) DO NOTHING — signature verification not yet wired.
  • POST /bookings/:id/confirm-payment-stub exists as the legacy "automatic-confirm" path; no longer auto-fired by the frontend.

Refunds

No dedicated table — refunds use the payments.refunded_amount accumulator (≤ amount) plus razorpay_refund_id / refund_reason / refunded_at. The recompute logic subtracts refunded_amount from the sum so booking balances stay correct.

08 · End-to-end user journeys

Narratives for each persona — exact pages, hooks, endpoints, and DB writes at every step.

Journey A · Patient books a test with home collection

Browse services on /services
ServicesPage → useServices() → GET /services → SELECT services.
Add 2–3 tests to cart
useCartStore.addItem() persists to localStorage arogya-cart.
Open /cart → click "Proceed"
Redirects to /book/test if signed in; otherwise /auth/login with state.from.
Sign up if needed
SignupPage → useAuthStore.signUp() → POST /auth/signup → INSERT users (role=patient, password_hash via bcrypt) → returns JWT.
BookTestPage step 2: choose home collection
Enter pincode → GET /serviceable-pincodes/check?pincode=. If serviceable, the home-visit charge appears and the slot grid unlocks.
Pick slot
useHomeCollectionSlots(date, pincode) → GET /home-collection/slots (currently STUB: 8 slots).
Step 3: confirm + pay
UpiPaymentModal renders QR (react-qr-code) from useClinicUpi(). Patient pays externally, uploads screenshot/PDF (≤5 MB) + optional UTR.
Submit booking (multipart)
useCreateTestBooking() → POST /bookings/test (FormData). Single transaction inserts bookings + N booking_items + invoices (trigger fills number) + payments (upi_manual, captured, verified_at=NULL).
Redirect to /payment/callback
PaymentCallbackPage loads useMyBooking(id), shows status "In progress · awaiting verification", Download Invoice button works (ownership-checked PDF).
Admin re-verifies later
AdminPaymentVerificationsPage → POST /admin/payments/:id/re-verify → status flips to "Confirmed", patient notification fires.
Sample collection day
Home Collection Board admin assigns a collector → collection_status transitions not_assigned → assigned → en_route → collected → received_at_lab; each transition fires a notification.
Result uploaded
Admin opens /admin/reports, types booking_code (AROGYA-TEST-…), uploads N report files. Patient sees them on /dashboard/reports and downloads via signed URL.

Journey B · Patient books a doctor appointment

Browse doctors on /doctors
DoctorsPage → useDoctors() → GET /doctors (active + verified) → photo via public /doctors/:id/photo blob.
Open doctor profile → "Book appointment"
Navigates to /book/doctor/:doctorId (requireAuth).
Step 1: pick date + slot
useDoctorSlots(id, date, centerId, visitType) → GET /doctors/:id/slots → live generator reads doctor_centers.schedule, subtracts existing bookings and doctor_unavailability.
Step 2: contact + address
Form prefilled from useAuthStore user; uses patientInfoSchema (zod).
Step 3: pay + upload proof
Same UpiPaymentModal pattern. POST /bookings/doctor-appointment (multipart) — atomic INSERT bookings + booking_items(doctor_consultation) + invoices + payments.
Status badge sub-line
BookingStatusBadge renders "Advance ₹250 paid · Balance ₹250 due at visit" instead of generic status text.

Journey C · Receptionist creates a walk-in bill

Open /admin/walk-in-bills/new
WalkInBillPage — patient snapshot + items[] (catalog | custom) + optional payment.
Submit
useCreateWalkInBill() → POST /admin/walk-in-bills. Single transaction does: resolve catalog → normalize → compute totals → INSERT bookings (origin=walk_in, visit_type=in_clinic, scheduled_date=today) → N × INSERT booking_items → INSERT payments (offline, captured) if amount > 0 → INSERT invoices (trigger fills number).
Hand patient the printed bill
Admin uses the Download Invoice button on the new booking; pdfkit streams an A4 PDF with clinic header from clinic_settings.

Journey D · Admin onboards a new doctor

Open /admin/doctors → "Add doctor"
AdminDoctorEditPage with WeeklyScheduleEditor.
Fill profile + add center(s) + schedule
Each row uses a checkbox-per-weekday + start/end + slot_minutes + buffer_minutes + optional lunch window → emits a WeeklySchedule JSON.
Submit
useCreateDoctor() → POST /admin/doctors (with nested centers[]). Transaction inserts users (role=doctor, is_login_enabled=FALSE) + N doctor_centers.
Upload photo
useUploadDoctorPhoto(id) → POST /admin/doctors/:id/photo (multer 5 MB). Bytes written to users.profile_photo_bytes; profile_photo_url set to /doctors/:id/photo.
Verify on public site
Public /doctors/:id renders the new doctor with ?v=updated_at cache-bust so the new photo loads.

Journey E · Admin uploads reports against any booking

Open /admin/reports
AdminReportsPage shows recent uploads. The page accepts free-text and a booking_code lookup.
Type AROGYA-TEST-… code
lookupBookingByCode() → GET /admin/booking-lookup?code= → resolves to numeric id + patient_snapshot.
Pick file(s), choose report_type
For each file: useUploadReport(id) → POST /admin/bookings/:id/reports (multer, 25 MB, no MIME whitelist).
Cache invalidation
Mutation invalidates adminBooking(id) + myReports + ['admin','reports'] — patient dashboard shows the new report on next refetch.

10 · Database connection graph (the FK story)

Five domains. Every table on a real flow above is here with its keys, indexes, and the relationships it joins on.

Domain map (high level)

users (STI) is the actor table. bookings is the hub: it FK-joins to users (patient / doctor / assigned_staff), to doctor_centers, and is the parent of booking_items, payments, invoices, reports. Catalog tables (services, service_categories, departments) reference each other and feed booking_items.service_id and users.department_id. Key triggers & constraints

  • generate_booking_code() — BEFORE INSERT on bookings; uses seq_booking_code_doc / seq_booking_code_test; month in Asia/Kolkata.
  • generate_invoice_number() — BEFORE INSERT on invoices; uses seq_invoice_number; month in Asia/Kolkata.
  • set_updated_at() — reused on every table with an updated_at column.
  • uq_active_slot — partial UNIQUE index that prevents double-booking while a slot is active.
  • services_package_consistency_chk — if is_package then package_service_ids must be a JSON array.
  • users_login_capability_chk — can't enable login without a way to authenticate.
  • payments_source_consistency_chk — UPI manual rows need proof bytes; offline rows need collected_by_user_id; Razorpay rows need razorpay_order_id.

11 · Cross-cutting concerns (every request)

Invisible plumbing that protects every endpoint.

Middleware chain

helmet → cors → json(1mb) → urlencoded → pinoHttp → generalLimiter(200/min) → static /uploads → auditLogMiddleware → router → notFoundHandler → errorHandler

Error translation

ZodError→400 · MulterError→413/400 · pg 23505→409 · 23503/02/14→400 · 42703/42P01→500 with "pnpm db:migrate" hint.

Transaction boundary

Every multi-table booking write uses transaction(fn) in db/pool.ts — BEGIN / COMMIT / ROLLBACK, client released in finally.

Cache invalidation rule

Mutations invalidate every affected key. useCreateServiceCategory invalidates 3 keys: admin list · public list · related dropdowns.

booking_code is canonical

One human-readable ID ties patient → booking → items → payments → invoice → reports. Admin always types the code.

Storage adapter

LocalStorageAdapter writes to STORAGE_LOCAL_PATH, signs via getSignedUrl(key, 900s). S3 driver falls back to local with a warn.

Rate limits

Global 200/min/IP via generalLimiter. PublicWriteLimiter 5/min on POST /enquiries and POST /reviews.

JWT freshness

requireAuth re-SELECTs the users row every request — role changes / disabled accounts take effect immediately.

Mock-mode flag

VITE_USE_MOCK_DATA !== 'false' short-circuits Phase-1 read hooks to in-memory fixtures via mockOrFetch().

Asia/Kolkata everywhere

Booking codes, invoice numbers, dashboard today/yesterday, PDF dates — all derived in Asia/Kolkata, never UTC.

Phase-2 gate

VITE_PHASE_2_ENABLED gates the entire patient + admin surface so the Phase-1 marketing site can ship alone.

Pino redactions

Authorization headers, password*, token fields are scrubbed before any line is written.
