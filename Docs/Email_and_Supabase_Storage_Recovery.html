Email + Supabase Storage — Recovery Plan

Captured during development on 17 May 2026. Use this document to re-implement the email (Resend) and file-storage (Supabase Storage) work when production-readiness time arrives — currently the client is testing on the existing Render + Netlify + Supabase stack and no live code changes have been merged.

Why this exists

Two production-readiness gates have been identified in the current build:

  1. Patient files (lab reports, payment proofs, doctor photos) are stored as Postgres BYTEA. Fine for the MVP, but at ~5 GB of file volume the relational DB starts feeling it. The Express backend already has a pluggable StorageAdapter interface designed exactly for this swap.
  2. Transactional email (OTP, booking confirmed, report-ready, payment re-verified) is not yet sent. The notify() helper writes to the notifications table only.

This document captures the implementation that was prototyped and then reverted. When the client is ready (after testing concludes), follow the steps below to re-apply it cleanly.

Constraints recorded:
  • Custom Express API stays the source of truth — no Supabase Auth, no Edge Functions.
  • Email only — no SMS. MSG91 is dropped from the plan. OTP and notifications go via Resend.
  • Nothing in the existing flow may break. Every change is feature-flagged behind env vars; default behaviour is unchanged.
  • Current production stack: Render (backend) + Netlify (frontend) + Supabase (DB). The plan assumes this stack.

4. Code changes — order to apply

4.1 Extend the storage adapter backend/src/lib/storage.ts

Add a download() method to the StorageAdapter interface, then add a SupabaseStorageAdapter class implementing it. Key implementation notes:

  • Use the tilde character (~) as the bucket/object separator inside storage_key, e.g. reports~abc-123.pdf. The tilde is URL-safe per RFC 3986 (no encodeURIComponent dance) and keeps the path parameter a single segment.
  • Map folder names to bucket env vars: reports → SUPABASE_BUCKET_REPORTS, payment-proofs → SUPABASE_BUCKET_PAYMENT_PROOFS, etc.
  • Build the public URL as ${BACKEND_PUBLIC_URL}/api/v1/files/${key}. The backend proxies the download from the bucket, so private buckets stay private and the frontend URL contract is unchanged.
  • The factory function buildStorage() picks the adapter based on STORAGE_DRIVER and falls back to LocalStorageAdapter if Supabase keys are missing — so a partial configuration never crashes the backend.
  • Export an isSupabaseStorageActive() helper for routes to check.

4.2 Build the Resend wrapper backend/src/lib/email.ts (new)

A thin, typed wrapper around resend.emails.send():

  • Lazy-init the Resend client (only when RESEND_API_KEY is set — so the file imports cleanly in dev without a key).
  • sendEmail({ to, subject, html, text, idempotencyKey, replyTo }) → returns { ok, id?, error?, skipped? }.
  • Skip silently if no key is set; log the OTP code to console in dev mode (mirrors today's mobile-OTP debug echo).
  • Redact recipients in logs as p***@example.com.
  • Pass X-Entity-Ref-ID header when an idempotency key is provided so duplicate sends are visible in Resend's dashboard.

4.4 Extend notify() backend/src/lib/notify.ts

Add an optional email field to NotifyArgs:

email?: {
  subject: string;
  html: string;
  text: string;
  idempotencyKey?: string;
  overrideTo?: string | string[];   // for admin-alert audience
}

After the DB insert, if email is present:

  1. Look up the recipient email by user_id from the users.email column (or use overrideTo).
  2. Call sendEmail() — best-effort, failure logs but never throws (matches the existing notify() contract).
  3. Email step happens outside the optional transaction — a failed Resend call never rolls back the underlying business action.

4.5 Wire emails at call sites 2 places, additive only

  • backend/src/modules/uploads/uploadRoutes.ts — when the admin uploads a report (event report_uploaded), render reportReady({ patientName, bookingCode }) and pass it to notify(). Need to LEFT JOIN users in the booking lookup to get the first name.
  • backend/src/modules/phase2/phase2Routes.ts — when the admin re-verifies a UPI payment in person (event reverified), render paymentReverified({ patientName, bookingCode, amount }) and pass it to notify(). The amount comes from payments.amount (NUMERIC, not paise).

4.6 Email-OTP endpoints backend/src/modules/auth/authRoutes.ts

Two new endpoints, parallel to the existing mobile-OTP flow — mobile-OTP keeps working, email-OTP is opt-in:

  • POST /auth/otp/email/send — accepts { email }, issues a 6-digit code via otpStore.issue('email:' + email.toLowerCase()), renders otpEmail() template, sends via Resend. Protected by otpIssueLimiter (5 per 10 min per IP — add to middleware/rateLimit.ts).
  • POST /auth/otp/email/verify — accepts { email, code, first_name?, last_name? }, verifies via otpStore.verify('email:' + email, code), auto-creates a patient account on first verify (with email-only, no mobile required — the users schema allows it).
OTP store namespacing. The store is keyed by identifier. Mobile flow uses the raw mobile number; email flow uses email:<lowercased-email>. The prefix prevents any chance of collision and makes the audit trail clear.

4.7 Refactor reports upload to write to Supabase uploadRoutes.ts

Dual-path: when isSupabaseStorageActive(), call storage.upload('reports', ...) and store the returned key in reports.storage_key with file_bytes = NULL. Otherwise, keep the existing BYTEA path. The download route (GET /files/:storageKey) checks file_bytes first:

if (row.file_bytes) {
  // legacy BYTEA row — serve inline
  res.end(row.file_bytes);
} else {
  // Supabase row — fetch from bucket
  const blob = await storage.download(row.storage_key);
  if (!blob) throw 404;
  res.end(blob.buffer);
}

This means no data migration is required — old rows keep working, new uploads go to Supabase. Migrate old rows later (or never).

Update the route's storage-key regex to allow ~: /^[a-zA-Z0-9._~\-]+$/.

4.8 Apply the same pattern to other upload routes later

Once reports is proven in production, repeat the dual-path swap for:

  • Doctor / collector profile photos (currently users.profile_photo_bytes — migration 0022)
  • Payment proof uploads (currently payments.payment_proof_bytes — migration 0024)

Same pattern: gate on isSupabaseStorageActive(), store the Supabase key, fall back to BYTEA for old rows.

5. Security checklist before going live

  • Service-role key only set in the Render backend env — never in the frontend, never committed.
  • HTTPS everywhere — Render gives this for free.
  • HSTS at the host layer.
  • OTP rate-limit active: 5 issues per 10 min per IP.
  • bcrypt rounds ≥ 12 in production (currently 10 default).
  • JWT secret ≥ 64 random chars — regenerate before go-live.
  • No PII in logs — already enforced by the redact() helper in email.ts.
  • Resend domain verified — DKIM, SPF, DMARC records in your registrar for arogyaclinic.in.
  • Buckets reports, payment-proofs, invoices are Private in Supabase Studio.

6. Activation order on the Render service

  1. Create the 4 buckets in Supabase Studio (must exist before STORAGE_DRIVER=supabase).
  2. Sign up at resend.com, grab the API key.
  3. Set all the env vars listed in §2 on the Render service (Settings → Environment).
  4. Deploy the code from §4. Backend restarts — Render handles this automatically.
  5. Smoke test: upload one report via admin UI → check it landed in the reports bucket in Supabase Studio. Open the patient's email — they should receive the report-ready notification.
  6. Optionally: write a one-off script to migrate historical BYTEA rows to Supabase Storage (read rows where file_bytes IS NOT NULL, upload via the adapter, set storage_key, NULL out file_bytes). Don't drop the BYTEA columns until backups confirm.

7. Rollback plan

If anything misbehaves after activation:

  1. On Render → Settings → Environment, set STORAGE_DRIVER=local (or unset it — default is local). Save → service restarts. New uploads go back to BYTEA, old Supabase uploads stay readable as long as the bucket still exists.
  2. To disable email entirely, blank out RESEND_API_KEY. The sendEmail() wrapper skips silently when no key is set — no errors thrown.
  3. Both gates above are independent — you can roll back one without affecting the other.
